# Context-Aware Documentation Discovery

**Feature**: Intelligent keyword-based documentation routing in the
Nightgauge pipeline.

**Issue**: #243 **Status**: Implemented **Version**: 1.9.0 (feature-planning
skill)

---

## Problem Statement

The Nightgauge pipeline's adaptive documentation reading (from Issue #162)
selects documentation scope based on issue size labels (XS/S/M/L/XL). However,
it doesn't consider the **content** of the issue when deciding which docs to
read.

This results in:

- **Missed relevance**: A security-focused bug might read architecture docs
  instead of security docs first
- **Unnecessary reads**: Simple git workflow questions read all documentation
- **No content awareness**: Two M-sized issues read the same docs regardless of
  topic

**Example**: Issue "Fix SQL injection in login endpoint" (size:S) currently
reads CODE_STANDARDS.md and TESTING.md (targeted scope), but should prioritize
SECURITY.md based on keywords.

---

## Solution Overview

Context-aware documentation discovery adds a **keyword extraction and matching
phase** before documentation reading:

```
┌─────────────────────────────────────────────────────────────────┐
│  PHASE 1.5: DOCUMENTATION DISCOVERY (Deterministic)             │
│  ├─ Step 1.5.1: Extract keywords from issue content             │
│  ├─ Step 1.5.2: Load Documentation Map from CLAUDE.md           │
│  ├─ Step 1.5.3: Match keywords to documentation topics          │
│  ├─ Step 1.5.4: Build prioritized doc reading list              │
│  └─ Step 1.5.5: Fall back to scope-based defaults if no match   │
└─────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│  PHASE 2: ADAPTIVE DOC READING (Modified)                       │
│  ├─ Use discovered doc list (minimal/targeted scope)            │
│  ├─ OR read all docs (standard/extended scope)                  │
│  └─ Track which docs were actually read                         │
└─────────────────────────────────────────────────────────────────┘
```

## Documentation Map Format

The Documentation Map is stored in CLAUDE.md as a Markdown table:

```markdown
## Documentation Map

> This map helps AI agents find relevant documentation based on the task at
> hand. Keywords are matched against issue content to prioritize which docs to
> read.

| Topic        | Primary Docs           | Keywords                                    |
| ------------ | ---------------------- | ------------------------------------------- |
| Architecture | docs/ARCHITECTURE.md   | architecture, design, components, structure |
| Git          | docs/GIT_WORKFLOW.md   | git, branch, commit, merge, pull, request   |
| Security     | docs/SECURITY.md       | security, validation, secrets, auth, input  |
| Testing      | docs/TESTING.md        | test, coverage, unit, integration, e2e      |
| Standards    | docs/CODE_STANDARDS.md | naming, style, format, convention, standard |
```

### Why CLAUDE.md?

- **Single source**: CLAUDE.md is the authoritative AI configuration file
- **Small footprint**: Map is ~10-15 lines, keeps CLAUDE.md concise
- **Generated by smart-setup**: Automatic creation during repository setup
- **Human-editable**: Teams can customize keyword mappings

---

## Keyword Extraction Algorithm

Keywords are extracted deterministically (zero LLM tokens) using pattern
matching:

```bash
# Extract from issue title, summary, and technical notes
ALL_TEXT="$ISSUE_TITLE $REQUIREMENTS_SUMMARY $TECHNICAL_NOTES"

# Extract 4+ character words, lowercase, deduplicated
KEYWORDS=$(echo "$ALL_TEXT" | tr '[:upper:]' '[:lower:]' | \
  grep -oE '\b[a-z]{4,}\b' | sort -u | tr '\n' ' ')
```

### Why 4+ Characters?

- Filters noise words (the, and, for, etc.)
- Captures meaningful terms (security, testing, branch)
- Balances precision and recall

### Why Deterministic?

Following the
[Deterministic vs Probabilistic architecture principle](ARCHITECTURE.md#deterministic-vs-probabilistic-architecture):

- **Zero LLM tokens** for keyword extraction
- **Predictable behavior** (same content = same keywords)
- **Fast execution** (<1ms)
- **Debuggable** (clear rules, no black box)

---

## Matching Algorithm

Keywords are matched against the Documentation Map using case-insensitive word
boundary matching:

```bash
for kw in $KEYWORDS; do
  if echo "$map_keywords" | grep -qiw "$kw"; then
    MATCHED_DOCS="$MATCHED_DOCS $docs"
  fi
done
```

### Match Priority

1. **Keyword matches** (highest priority) - Docs matching issue keywords
2. **Essential docs** (always included) - GIT_WORKFLOW.md
3. **Scope defaults** (fallback) - Standard scope-based reading

---

## Discovery Methods

The `docs_consulted.discovery_method` field tracks how documentation was
selected:

| Method            | Description                                   | When Used                      |
| ----------------- | --------------------------------------------- | ------------------------------ |
| `keyword-matched` | Keywords from issue matched Documentation Map | Keywords found matches         |
| `scope-fallback`  | No keyword matches, used scope-based defaults | No matches, non-extended       |
| `extended-all`    | Extended scope reads all docs regardless      | size:L/XL or priority:critical |

## Token Efficiency Impact

### Before (Adaptive Reading Only)

| Issue Type     | Scope    | Docs Read Based On |
| -------------- | -------- | ------------------ |
| XS bug         | Minimal  | Size label         |
| S security bug | Targeted | Size label         |
| M feature      | Standard | Size label         |

### After (With Discovery)

| Issue Type     | Scope    | Docs Read Based On           | Additional Savings |
| -------------- | -------- | ---------------------------- | ------------------ |
| XS bug         | Minimal  | Size + keywords              | ~10%               |
| S security bug | Targeted | Keywords → SECURITY.md first | ~30%               |
| M feature      | Standard | Keywords prioritize reading  | ~20%               |

**Conservative estimate**: 20-40% additional token savings for minimal/targeted
scope issues when keywords match.

## Backward Compatibility

- **No breaking changes**: `docs_consulted` is optional
- **Default behavior preserved**: Missing Documentation Map falls back to
  scope-based reading
- **Graceful degradation**: Repositories without CLAUDE.md continue working
- **Schema version bump**: 1.0 → 1.1 (additive change)

## Integration with Adaptive Reading

Context-aware discovery integrates with the existing adaptive documentation
reading feature (Issue #162):

```
┌─────────────────────────────────────────────────────────────────┐
│  Phase 0.5: Complexity Assessment (existing)                    │
│  └─ Determines DOC_SCOPE (minimal/targeted/standard/extended)   │
└─────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│  Phase 1.5: Documentation Discovery (NEW)                       │
│  ├─ Extract keywords from issue                                 │
│  ├─ Match against Documentation Map                             │
│  └─ Build PRIORITY_DOCS list                                    │
└─────────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────────┐
│  Phase 2: Adaptive Documentation Reading (modified)             │
│  ├─ If PRIORITY_DOCS set: read matched docs first               │
│  ├─ Else: use scope-based defaults                              │
│  └─ Extended scope ignores discovery (reads all)                │
└─────────────────────────────────────────────────────────────────┘
```

## Configuration

No configuration required. The feature works automatically when:

1. CLAUDE.md exists with a Documentation Map section
2. Issue content contains extractable keywords

### Customizing the Documentation Map

Teams can customize keyword mappings by editing the Documentation Map in
CLAUDE.md:

```markdown
| Topic    | Primary Docs     | Keywords                         |
| -------- | ---------------- | -------------------------------- |
| Database | docs/DATABASE.md | database, postgres, mysql, query |
| API      | docs/API.md      | api, endpoint, rest, graphql     |
```

## Real Workflow Example

Below is a concrete walkthrough showing how context-aware discovery selects
documentation for an actual pipeline run.

**Issue**: `#1403 — Fix OutputWindow stealing focus from editor`

**Step 1 — Keyword extraction** from issue title and body:

```
Extracted keywords: [output, window, focus, editor, preserve, reveal, panel]
```

**Step 2 — Documentation map matching** against the CLAUDE.md table:

```
Match results (scored by keyword overlap):
  1. docs/ARCHITECTURE.md          — keywords: [components, structure]    → score: 0 (no overlap)
  2. docs/CONFIGURATION.md         — keywords: [config, settings]         → score: 0
  3. docs/PIPELINE_EXECUTION.md    — keywords: [execution, mode]          → score: 0
  ...
```

No high-confidence matches from the documentation map — this is a focused VSCode
extension bug, not an architectural concern.

**Step 3 — Complexity-aware scope selection** (via adaptive reading):

```
Issue labels: [type:fix, size:XS, priority:high]
Complexity assessment: XS (bug fix)
Documentation scope: minimal
```

**Step 4 — Final prioritized document list**:

```
Documents to read (minimal scope):
  1. docs/GIT_WORKFLOW.md           — always included for commit conventions
  2. standards/security.md          — always included for input validation
Total: 2 documents (~1,500 tokens)
```

The pipeline skipped all architecture and feature docs because the XS bug fix
only needed commit conventions and security basics. A larger feature issue (M+)
would have matched broader keywords and included architecture and testing docs.
