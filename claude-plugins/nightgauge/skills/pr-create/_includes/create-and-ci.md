## Phase 3: create PR via Go binary

Create the PR using the Go binary. The skill constructs the title, body (with
correct closing keywords), and handles reviewer assignment with self-reviewer
guard. Append `KNOWLEDGE_SECTION` to the PR body when non-empty:

```bash
BINARY="${NIGHTGAUGE_BIN:-}"
[ -n "$BINARY" ] && [ ! -x "$BINARY" ] && BINARY=""
[ -z "$BINARY" ] && BINARY=$(command -v nightgauge 2>/dev/null || echo "")
if [ -z "$BINARY" ]; then
  REPO_ROOT="$(git rev-parse --show-toplevel 2>/dev/null || pwd)"
  [ -x "$REPO_ROOT/bin/nightgauge" ] && BINARY="$REPO_ROOT/bin/nightgauge"
fi
if [ -z "$BINARY" ]; then
  GIT_COMMON_DIR="$(git rev-parse --git-common-dir 2>/dev/null || true)"
  if [ -n "$GIT_COMMON_DIR" ]; then
    CANONICAL_REPO="$(cd "$GIT_COMMON_DIR/.." 2>/dev/null && pwd)"
    [ -n "$CANONICAL_REPO" ] && [ -x "$CANONICAL_REPO/bin/nightgauge" ] && BINARY="$CANONICAL_REPO/bin/nightgauge"
  fi
fi
[ -z "$BINARY" ] && [ -x "$HOME/go/bin/nightgauge" ] && BINARY="$HOME/go/bin/nightgauge"
[ -n "$BINARY" ] && export PATH="$(dirname "$BINARY"):$PATH"

# Push the branch idempotently. feature-dev usually already pushed it, so a
# rejected push (diverged worktree) is NOT fatal — if origin already has the
# branch, the work is on the remote and we open the PR from it. NEVER force-push
# (the safety hook blocks it) and NEVER fall back to asking the user: that
# dead-ends in headless mode (#3804). Mirrors the deterministic path (#3828/#3835).
if ! git push -u origin "$BRANCH_NAME" 2>&1; then
  if git ls-remote --exit-code --heads origin "$BRANCH_NAME" >/dev/null 2>&1; then
    echo "Push rejected but origin already has $BRANCH_NAME — proceeding to PR creation from the remote branch"
  else
    echo "ERROR: Failed to push branch $BRANCH_NAME to origin and it does not exist on the remote" >&2
    echo "Do NOT force-push and do NOT ask for input — report this as a terminal pr-create failure with the push error above." >&2
    exit 1
  fi
fi

RESULT=$("$BINARY" pr create \
  --title "$PR_TITLE" \
  --body "$PR_BODY" \
  --head "$BRANCH_NAME" \
  --base "$BASE_BRANCH" \
  --json 2>&1) || {
  echo "ERROR: PR creation failed: $RESULT" >&2
  exit 1
}
```

If batch mode was detected in Phase 1.5, use the epic branch and batch PR body:

```bash
RESULT=$("$BINARY" pr create \
  --title "$BATCH_PR_TITLE" \
  --body "$BATCH_PR_BODY" \
  --head "$EPIC_BRANCH" \
  --base main \
  --json 2>&1) || {
  echo "ERROR: Batch PR creation failed: $RESULT" >&2
  exit 1
}
```

The skill handles:

- PR title construction with correct prefix (feat/fix/docs/refactor/chore)
- PR body with correct closing keywords (`Part of #PARENT` + `Closes #N` for
  epic sub-issues so the sub-issue auto-closes on merge, `Closes #N` for
  standalone issues)
- Validation summary and dead code sections from validate context
- `## What to Test` section appended after `## Validation` when
  `WHAT_TO_TEST_SECTION` is non-empty
- `## Knowledge` section appended when `KNOWLEDGE_SECTION` is set
- Reviewer assignment with self-reviewer guard
- Branch push before PR creation

**Single-issue PR body structure** (for reference — skill generates this):

```markdown
## Summary

[Concise description of changes]

## Changes

- [List of key changes]

## Validation

- Build: Passed / Skipped
- Tests: X passed, 0 failed

## What to Test

(generated by WhatToTestGenerator when dependency graph is available)

## Knowledge

- [PRD](.nightgauge/knowledge/features/1234-slug/PRD.md) — Product
  requirements and design decisions
- [Decisions](.nightgauge/knowledge/features/1234-slug/decisions.md) —
  Architecture and implementation decisions

Part of #PARENT
Closes #N
```

For standalone issues (no epic parent), omit the `Part of` line. For epic
sub-issues, include BOTH lines so the sub-issue auto-closes on merge.

When a single PR delivers **more than one** of an epic's sub-issue deliverables
(an epic-umbrella PR), enumerate a `Closes #sub` line for **every** sub whose
work is in the diff — never close only the epic. Closing just `#epic` leaves the
subs OPEN and the autonomous picker re-spawns them as conflicting PRs (#3979).
The epic then auto-closes natively once all its subs close.

The `## What to Test` section is included only when `WHAT_TO_TEST_SECTION` is
non-empty (i.e., the dependency graph exists and the diff produced results). The
`## Knowledge` section is included only when `KNOWLEDGE_SECTION` is non-empty
(i.e., knowledge entries exist from the pipeline run). When a section is absent,
it is omitted entirely — never include an empty section.

Assign reviewers after PR creation (skip if reviewer is the PR author):

```bash
if [ -n "$REVIEWER" ] && [ "$REVIEWER" != "$PR_AUTHOR" ]; then
  # Add reviewer via GitHub REST API (no forge CLI dependency)
  OWNER=$(echo "$REPO" | cut -d'/' -f1)
  REPO_NAME=$(echo "$REPO" | cut -d'/' -f2)
  curl -s -X POST \
    -H "Authorization: Bearer ${GITHUB_TOKEN}" \
    -H "Accept: application/vnd.github+json" \
    -H "X-GitHub-Api-Version: 2022-11-28" \
    "https://api.github.com/repos/${OWNER}/${REPO_NAME}/pulls/${PR_NUMBER}/requested_reviewers" \
    -d "{\"reviewers\":[\"${REVIEWER}\"]}" > /dev/null 2>&1 || \
    echo "WARNING: Could not assign reviewer $REVIEWER" >&2
fi
```

Extract results (field names match the Go `PullRequest` struct):

```bash
PR_NUMBER=$(echo "$RESULT" | jq -r '.number')
PR_URL=$(echo "$RESULT" | jq -r '.url')
PR_TITLE=$(echo "$RESULT" | jq -r '.title')
PR_BASE_BRANCH=$(echo "$RESULT" | jq -r '.baseRef')
```

## Phase 3.6: verify PR created

**PURPOSE**: Post-condition verification that a PR actually exists on GitHub for
the feature branch. Protects against silent pr-create failures where the subagent
exits cleanly but never invokes `nightgauge forge pr create` (or the Go binary's `pr create`
command).

**Step 3.6.1: Query GitHub for PR on this branch**

```bash
PR_CHECK=$("$BINARY" forge pr list --repo "$REPO" \
  --head "$BRANCH" \
  --state all \
  --json number,state,url \
  2>/dev/null || echo "[]")

PR_EXISTS=$(echo "$PR_CHECK" | jq 'length' 2>/dev/null || echo "0")
```

**Step 3.6.2: Cross-check with `gh` before declaring not-found**

`forge pr list --head` intermittently returns `[]` for a genuinely-open PR. Do
NOT trust a single empty result — cross-check with `gh` directly (which queries
the live GitHub API by head ref) before concluding the PR is missing:

```bash
if [ "$PR_EXISTS" -eq 0 ]; then
  # Second opinion: gh queries the API directly by head ref.
  GH_CHECK=$(gh pr list --repo "$REPO" --head "$BRANCH" --state all \
    --json number,state,url 2>/dev/null || echo "[]")
  PR_EXISTS=$(echo "$GH_CHECK" | jq 'length' 2>/dev/null || echo "0")
  [ "$PR_EXISTS" -gt 0 ] && PR_CHECK="$GH_CHECK"
fi
```

**Step 3.6.3: Handle PR genuinely not found**

If BOTH `forge` and `gh` agree no PR exists, the PR was never created — this is a
hard failure. Do not paper over it:

```bash
if [ "$PR_EXISTS" -eq 0 ]; then
  echo ""
  echo "ERROR: PR creation reported success but no PR found on GitHub for branch: $BRANCH"
  echo "  (confirmed via both 'forge pr list' and 'gh pr list')"
  echo ""
  echo "Troubleshooting:"
  echo "  1. Check branch exists: git branch -a | grep $BRANCH"
  echo "  2. Check branch is pushed: git push -u origin $BRANCH"
  echo "  3. Re-run pr-create stage to attempt again"
  echo "  4. Or create PR manually:"
  echo "       nightgauge forge pr create --title \"[...title...]\" --body \"[...body...]\" --head $BRANCH --base $BASE_BRANCH"
  echo ""
  exit 1
fi
```

**Result**: If PR was found (by either tool), continue to Phase 3.5 normally. If
not found, stage exits with status 1, and the orchestrator classifies this as
`errorCategory: "pr-not-created"`.

> **SUCCESS CONTRACT (HARD RULE).** pr-create may only report success and write
> `pr-{N}.json` when this phase has confirmed an OPEN PR with a real `PR_NUMBER`.
> If pushing the branch or creating the PR was blocked (e.g. a destructive-git
> hook rejected the push, an `AskUserQuestion` was dismissed in autonomous mode,
> or the branch never reached the remote), you MUST exit non-zero — do NOT write
> a context/assessment file that claims the PR was created. A clean exit with no
> PR is a false success that strands the issue and pages the operator. When in
> doubt, fail loudly: pr-create is idempotent and safe to re-run.

## Phase 3.5: monitor CI status

**Activation**: Runs when `PR_NUMBER` and `BINARY` are available from Phase 3.
Skips gracefully (sets `CI_MONITORED=false`) if either is absent.

**Headless safe**: No interactive prompts. All output is informational only.

### Step 3.5.1: Configure and Run CI Wait

```bash
CI_TIMEOUT=${NIGHTGAUGE_CI_TIMEOUT_MINS:-15}
CI_POLL=${NIGHTGAUGE_CI_POLL_SECS:-30}
CI_MONITORED=false
CI_FINAL_STATUS="pending"
CI_CHECKS_TOTAL=0
CI_CHECKS_PASSED=0
CI_CHECKS_FAILED=0
CI_CHECKS_PENDING=0
CI_MONITOR_DURATION=0
CI_FAILURES_JSON="[]"
CI_NOTES=""
CI_MONITOR_TIMESTAMP=""

if [ -z "$BINARY" ] || [ -z "$PR_NUMBER" ] || [ "$PR_NUMBER" = "null" ]; then
  echo "CI monitoring: skipped (binary or PR number unavailable)"
else
  echo "Monitoring CI checks for PR #${PR_NUMBER} (timeout: ${CI_TIMEOUT}m, poll: ${CI_POLL}s)..."

  CI_RESULT=$("$BINARY" ci wait "$PR_NUMBER" \
    --timeout "$CI_TIMEOUT" \
    --poll "$CI_POLL" \
    --json 2>&1) || true

  CI_MONITOR_TIMESTAMP=$(date -u +%Y-%m-%dT%H:%M:%SZ)

  # Parse result fields from Go binary JSON output
  # Fields: state, total, successful, failed, pending, elapsedSecs, checks[]
  CI_STATE=$(echo "$CI_RESULT" | jq -r '.state // "UNKNOWN"' 2>/dev/null || echo "UNKNOWN")
  CI_CHECKS_TOTAL=$(echo "$CI_RESULT" | jq -r '.total // 0' 2>/dev/null || echo "0")
  CI_CHECKS_PASSED=$(echo "$CI_RESULT" | jq -r '.successful // 0' 2>/dev/null || echo "0")
  CI_CHECKS_FAILED=$(echo "$CI_RESULT" | jq -r '.failed // 0' 2>/dev/null || echo "0")
  CI_CHECKS_PENDING=$(echo "$CI_RESULT" | jq -r '.pending // 0' 2>/dev/null || echo "0")
  CI_MONITOR_DURATION=$(echo "$CI_RESULT" | jq -r '.elapsedSecs // 0' 2>/dev/null || echo "0")

  CI_MONITORED=true

  # Fallback (#273 defense-in-depth, mirrors the Phase 3.6 forge/gh
  # cross-check pattern): `ci wait` now populates checks[]/total on a
  # terminal FAILURE/ERROR verdict itself, but if that in-process
  # augmentation couldn't complete (e.g. a transient REST error fetching
  # individual check runs), fall back to `gh pr view --json
  # statusCheckRollup` so classification never operates on an empty
  # evidence set. GitHub's rollup entries already use the same
  # name/status/conclusion shape Step 3.5.2 expects.
  if [ "$CI_CHECKS_TOTAL" -eq 0 ] && { [ "$CI_STATE" = "FAILURE" ] || [ "$CI_STATE" = "ERROR" ]; }; then
    GH_ROLLUP=$(gh pr view "$PR_NUMBER" --repo "$REPO" --json statusCheckRollup 2>/dev/null || echo '{"statusCheckRollup":[]}')
    GH_CHECKS=$(echo "$GH_ROLLUP" | jq -c '[.statusCheckRollup[]? | {name, status, conclusion, detailsUrl}]' 2>/dev/null || echo "[]")
    GH_CHECKS_COUNT=$(echo "$GH_CHECKS" | jq 'length' 2>/dev/null || echo "0")
    if [ "$GH_CHECKS_COUNT" -gt 0 ]; then
      CI_RESULT=$(echo "$CI_RESULT" | jq --argjson checks "$GH_CHECKS" '. + {checks: $checks, total: ($checks | length)}' 2>/dev/null || echo "$CI_RESULT")
      CI_CHECKS_TOTAL="$GH_CHECKS_COUNT"
      echo "  (ci wait returned no check detail for a $CI_STATE verdict — cross-checked via 'gh pr view --json statusCheckRollup': $GH_CHECKS_COUNT checks found)"
    fi
  fi
fi
```

### Step 3.5.2: Classify Failures

When `CI_STATE` is `FAILURE` or `ERROR`, extract failed checks and classify each
using the same patterns as `ciCheckHelpers.ts` (deterministic — no AI inference).
Note: `detailsUrl` is not yet available from the Go binary `ci wait` output — it
will be empty until the CheckDetail struct is extended in a future issue.

```bash
if [ "$CI_MONITORED" = "true" ] && { [ "$CI_STATE" = "FAILURE" ] || [ "$CI_STATE" = "ERROR" ]; }; then
  # Extract failed checks from Go binary result
  FAILED_CHECKS=$(echo "$CI_RESULT" | jq -c '[
    .checks[] |
    select(
      .status == "COMPLETED" and
      (.conclusion == "FAILURE" or .conclusion == "TIMED_OUT" or .conclusion == "ACTION_REQUIRED")
    )
  ]' 2>/dev/null || echo "[]")

  # Classify each failure using patterns matching ciCheckHelpers FAILURE_TYPE_PATTERNS
  CI_FAILURES_JSON=$(echo "$FAILED_CHECKS" | jq '
    map(
      . as $check |
      ($check.conclusion | ascii_downcase) as $conclusion |
      ($check.name | ascii_downcase) as $name |
      # Priority order: lint → test → build → typecheck → security → format → unknown
      (if   ($name | test("lint|eslint|pylint|rubocop|flake8|style|code.?quality"; "i")) then "lint"
       elif ($name | test("test|spec|vitest|jest|pytest|rspec|unit|e2e|integration|coverage"; "i")) then "test"
       elif ($name | test("build|compile|bundle|webpack|vite|esbuild|rollup|package"; "i")) then "build"
       elif ($name | test("type.?check|tsc|typescript|types?|mypy"; "i")) then "typecheck"
       elif ($name | test("security|audit|codeql|snyk|dependabot|vulnerability|sast|dast"; "i")) then "security"
       elif ($name | test("format|prettier|black|formatting"; "i")) then "format"
       else "unknown"
       end) as $failure_type |
      # Transient: timed_out conclusion or network/retry patterns in name
      (($conclusion == "timed_out") or
       ($name | test("timed?.?out|timeout|flaky|network.?error|connection.?refused|econnreset|enotfound|rate.?limit|503|502|504|retry"; "i"))
      ) as $is_transient |
      # detailsUrl: Go binary CheckDetail does not yet carry this field (future work)
      (.detailsUrl // "") as $details_url |
      {
        name: $check.name,
        failure_type: $failure_type,
        is_transient: $is_transient,
        conclusion: $conclusion,
        details_url: $details_url
      }
    )
  ' 2>/dev/null || echo "[]")
fi
```

### Step 3.5.3: Map Go Binary State to Final Status

```bash
case "$CI_STATE" in
  SUCCESS) CI_FINAL_STATUS="success" ;;
  FAILURE) CI_FINAL_STATUS="failure" ;;
  ERROR)   CI_FINAL_STATUS="error"   ;;
  TIMEOUT) CI_FINAL_STATUS="timeout" ;;
  PENDING) CI_FINAL_STATUS="pending" ;;
  *)       CI_FINAL_STATUS="unknown" ;;
esac
```

### Step 3.5.4: Report Results

```bash
echo ""
echo "=== CI Status Report ==="
echo "PR #${PR_NUMBER}: ${CI_FINAL_STATUS} (${CI_MONITOR_DURATION}s elapsed)"
echo "Checks: ${CI_CHECKS_TOTAL} total | ${CI_CHECKS_PASSED} passed | ${CI_CHECKS_FAILED} failed | ${CI_CHECKS_PENDING} pending"

if [ "$CI_FINAL_STATUS" = "failure" ] || [ "$CI_FINAL_STATUS" = "error" ]; then
  echo ""
  echo "Failed checks:"
  echo "$CI_FAILURES_JSON" | jq -r '.[] |
    "  [\(.failure_type)] \(.name)" +
    (if .is_transient then " (transient)" else "" end) +
    (if .details_url != "" then "\n    Logs: \(.details_url)" else "" end)
  ' 2>/dev/null || true

  # Offer quick fixes for auto-fixable types (format, lint)
  AUTO_FIXABLE=$(echo "$CI_FAILURES_JSON" | \
    jq '[.[] | select(.failure_type == "format" or .failure_type == "lint")] | length' \
    2>/dev/null || echo "0")

  if [ "$AUTO_FIXABLE" -gt 0 ]; then
    echo ""
    echo "Auto-fixable failures detected (handoff to pr-merge — do NOT fix here):"
    echo "$CI_FAILURES_JSON" | jq -r \
      '.[] | select(.failure_type == "format" or .failure_type == "lint") |
       "  [\(.failure_type)] \(.name)"' 2>/dev/null || true
    CI_NOTES="${AUTO_FIXABLE} auto-fixable CI failure(s) (format/lint) — pr-merge will resolve."
  fi

  echo ""
  echo "NOTE: pr-merge will apply its RALPH Loop auto-fix for any remaining failures."
  echo "      pr-create exits here — see Phase 3.5 HARD RULE (#3666)."

elif [ "$CI_FINAL_STATUS" = "timeout" ]; then
  echo "CI checks did not complete within ${CI_TIMEOUT} minutes."
  echo "pr-merge will continue monitoring CI status."
  CI_NOTES="CI monitoring timed out after ${CI_TIMEOUT} minutes. pr-merge will continue monitoring."

elif [ "$CI_FINAL_STATUS" = "success" ]; then
  echo "All CI checks passed. PR is ready for review."

elif [ "$CI_FINAL_STATUS" = "pending" ] || [ "$CI_MONITORED" = "false" ]; then
  echo "CI checks not yet available — pr-merge will monitor status."
fi
```

### Step 3.5.5: Variables Passed to Phase 4

```
CI_CHECKS_PASSED        — passed check count

```
