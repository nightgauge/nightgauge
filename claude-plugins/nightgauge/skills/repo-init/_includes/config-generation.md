## Phase 6: Generate .nightgauge/config.yaml

Create the directory structure and pipeline artifact `.gitignore`:

```bash
mkdir -p .nightgauge/pipeline/history .nightgauge/plans .nightgauge/logs .nightgauge/analysis
```

Create `.gitkeep` files so the directory structure is preserved in git:

```bash
for dir in .nightgauge/pipeline/history .nightgauge/plans .nightgauge/logs; do
  [ -f "$dir/.gitkeep" ] || touch "$dir/.gitkeep"
done
```

Create `.nightgauge/.gitignore` to prevent pipeline artifacts from
polluting `git status`. If the file already exists and contains
`nightgauge-gitignore-version:`, skip this step (the extension manages it
automatically). Otherwise, write the full file:

If `.nightgauge/config.yaml` already exists, ask:

```json
{
  "questions": [
    {
      "question": ".nightgauge/config.yaml already exists. How should we handle it?",
      "header": "Existing Config",
      "multiSelect": false,
      "options": [
        {
          "label": "Update field IDs only (Recommended)",
          "description": "Preserve existing pipeline settings, refresh project field IDs"
        },
        {
          "label": "Replace with fresh config",
          "description": "Overwrite with the standard template (your custom settings will be lost)"
        },
        {
          "label": "Skip config update",
          "description": "Leave the file as-is"
        }
      ]
    }
  ]
}
```

**If "Update field IDs only"**: Use Edit tool to update only the `project:`
block.

**If "Replace" or new file**: Generate the canonical template via the Go
binary's deterministic `config init` verb (audit row B10). The verb queries
the project board once via the resolved `--project` number and substitutes
all field/option IDs into the template, eliminating the heredoc-and-jq
discovery flow this skill used to maintain inline:

```bash
# Online mode — populates field IDs from GitHub in one call.
# OWNER, OWNER_TYPE, REPO_NAME, PROJECT_NUMBER were resolved in Phase 1/4.
nightgauge config init \
  --owner "$OWNER" \
  --owner-type "$OWNER_TYPE" \
  --repo "$REPO_NAME" \
  --project "$PROJECT_NUMBER" \
  --out .nightgauge/config.yaml \
  ${REPLACE_EXISTING:+--force}
```

`${REPLACE_EXISTING:+--force}` is set to `--force` only when the user picked
"Replace with fresh config" above. New-file invocations omit it; the verb
exits 1 without overwriting if the file already exists.

If the project board is unreachable (offline scaffolding, dev container
without forge auth), pass `--no-fetch` to emit the same template with
`<PROJECT_NUMBER>` / `<*_OPTION_ID>` placeholders intact. The user is then
expected to fill the IDs in manually:

```bash
nightgauge config init \
  --owner "$OWNER" \
  --owner-type "$OWNER_TYPE" \
  --repo "$REPO_NAME" \
  --project "$PROJECT_NUMBER" \
  --no-fetch \
  --out .nightgauge/config.yaml
```

The verb owns the template's exact shape (header, ordering, comments, sprint
sub-block, pipeline / sanitization / human-in-the-loop / knowledge blocks).
Skill prose handles only the surrounding "How should we handle it?" prompt
and the `.gitignore` block above. Do **not** re-emit a YAML heredoc here.

If `nightgauge config init` is unavailable (binary not on PATH for any
reason), fall back to the canonical template at
[docs/GO_BINARY.md#configuration](../../../docs/GO_BINARY.md#configuration);
copy that block verbatim and replace each `<PLACEHOLDER>` with the IDs
discovered in Phase 4.

**Note:** The root `.gitignore` no longer needs modification — all pipeline
artifact patterns are in `.nightgauge/.gitignore` (created above, and
auto-managed by the extension on activation).

## Phase 6.5: Verify field IDs in config.yaml

The pipeline's `projectFieldWriter.ts` reads field IDs from the `project.fields`
section of `config.yaml` to sync issue statuses on the project board during
stage transitions (e.g., "In progress" → "In review" → "Done"). **Without
`project.id` and `project.fields` in config.yaml, every pipeline stage will warn
"No project configuration found" and skip board status sync.**

Verify that the `config.yaml` written in Phase 6 includes these fields under the
`project:` key (they should already be there from Phase 6 generation):

```yaml
project:
  number: <PROJECT_NUMBER>
  id: <PROJECT_ID>
  fields:
    status:
      id: <STATUS_FIELD_ID>
      options:
        backlog: <BACKLOG_OPTION_ID>
        ready: <READY_OPTION_ID>
        in-progress: <IN_PROGRESS_OPTION_ID>
        in-review: <IN_REVIEW_OPTION_ID>
        done: <DONE_OPTION_ID>
    priority:
      id: <PRIORITY_FIELD_ID>
      options:
        p0: <P0_OPTION_ID>
        p1: <P1_OPTION_ID>
        p2: <P2_OPTION_ID>
    size:
      id: <SIZE_FIELD_ID>
      options:
        xs: <XS_OPTION_ID>
        s: <S_OPTION_ID>
        m: <M_OPTION_ID>
        l: <L_OPTION_ID>
        xl: <XL_OPTION_ID>
```

Option keys use lowercase kebab-case. The pipeline remaps them to title-case
at runtime (e.g., `in-progress` → `In progress`).

**Do NOT generate a separate `project-field-mappings.json` file.** All field
data lives in `config.yaml` as the single source of truth.

If a legacy `project-field-mappings.json` exists, delete it.
